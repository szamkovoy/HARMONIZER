# Регламент структуры проекта Harmonizer

## 1. Два слоя репозитория

| Слой | Назначение | Технологии |
|------|------------|------------|
| **Клиент (основной)** | Нативное мобильное приложение | **React Native**, **Expo SDK 54**, **Expo Router** (файловый роутинг в `app/`), TypeScript |
| **Сервер / API (архив веба)** | Прокси к Google AI (Gemini), без утечки ключей на устройство | Next.js App Router в **`_legacy_web/`**, деплой на **Vercel**; маршрут `POST /api/communicator` и др. |

Мобильный клиент не вызывает Gemini напрямую: запросы идут на развёрнутый origin (`EXPO_PUBLIC_COMMUNICATOR_API_URL` + `/api/communicator`). Папка `_legacy_web` сохранена как эталон серверной логики и как деплойный артефакт API; при необходимости тот же контракт можно вынести в отдельный serverless-слой, не меняя контракт клиента.

## 2. Корень репозитория (клиент Expo)

- `app/` — экраны и навигация (Expo Router).
- `components/` — общие UI-примитивы и обвязка шаблона Expo.
- `modules/` — функциональные модули продукта (см. ниже).
- `services/` — HTTP-клиенты к API, Supabase и др. (вызываются из `modules/` и экранов).
- `assets/` — шрифты, изображения, **иконки UI** (`assets/icons/`), подключаемые через `require()` / `Image`.

Исключено из актуальной сборки клиента: PWA, `manifest.json`, service workers, веб-специфичные пути `public/` в корне (актуальные статики для веба при `expo start --web` — через `assets/` и конфигурацию Expo).

## 3. Глобальная организация модулей

Все функциональные блоки приложения располагаются в **`/modules/`**. Каждый модуль — автономный «чёрный ящик» с явным публичным API (Props / функции).

## 4. Внутренняя структура модуля

Каждый модуль (например, `/modules/communicator/`) содержит:

- `ui/` — компоненты React Native (`View`, `Text`, `Pressable`, …).
- `core/` — логика, парсинг, типы, хуки без привязки к DOM.
- `api/` — обвязка стриминга и вызовов, опирающаяся на `services/`.
- `store/` — локальное состояние сессии (по мере необходимости).
- `readme.md` — входы, выходы, зависимости, статус.

## 5. Документация

- Общие концепции и дорожные карты: **`/docs/`** (в т.ч. [`roadmap.md`](roadmap.md), [`tech_stack.md`](tech_stack.md)).
- Спецификации модулей: `modules/<name>/readme.md` и при необходимости краткий указатель в `docs/modules/`.
- При значимых изменениях интерфейсов модулей обновляйте соответствующий `readme.md`.

## 6. Взаимодействие

Модули общаются через чёткие интерфейсы (Props, события). Прямой доступ во внутренние папки другого модуля (кроме экспортируемого API) не используется.

## 7. Текущие модули (каталог)

- **COMMUNICATOR** — [`modules/communicator/readme.md`](../modules/communicator/readme.md): чат с ИИ (голос через **expo-av**, текст), стриминг ответа, задел под Hume (`onEmotionSegment`), интеграция с будущим ASSISTANT.

## 8. Мобильная специфика (Expo)

- Запись звука: **`expo-av`** (`Audio.Recording`), разрешения микрофона в **`app.json`** (плагин `expo-av`, `NSMicrophoneUsageDescription` на iOS, запись на Android через разрешения, выдаваемые при запросе).
- Целевые параметры записи для совместимости с Hume и документацией: **16 kHz, mono** — см. [`hume_integration.md`](hume_integration.md); при необходимости настройка битрейта/пресетов в коде записи.
