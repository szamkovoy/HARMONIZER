# HRV по пальцевому PPG: полный алгоритм RMSSD и индекса стресса (Баевский)

Документ описывает **фактическую** реализацию в репозитории на момент последней правки. Он предназначен для внешней проверки (в т.ч. другой моделью): в разделах про **границы RR**, **два разных буфера ударов**, **тиры по числу валидных ударов** и **расхождение «сырого сегмента» и отображения** зафиксированы места, где чаще всего скрываются нестыковки.

**Код:** `modules/biofeedback/core/finger-analysis.ts` (пайплайн, калибровка, UI-поля), `modules/biofeedback/core/metrics.ts` (метрики практики), `modules/biofeedback/core/hrv-practice-constants.ts` (единые пороги: RR, числа ударов), `modules/biofeedback/core/ppg-bandpass.ts`, UI пробы: `modules/biofeedback/ui/BiofeedbackProbeScreen.tsx`. Интеграция с мандалой: `modules/biofeedback/core/mandala-adapter.ts` (нормализация в контракт другого модуля).

| Компонент | Роль |

## 1. Вход: от кадра камеры до точки анализа

1. Нативный слой (или симуляция) формирует сэмпл с **монотонным** `timestampMs` и статистикой ROI.
2. В анализаторе считается **оптический индекс** (как в коде):  
   `opticalValue = redMean − 0.35·greenMean − 0.15·blueMean`.
3. **Детренд:** по скользящему окну вычитается медиана; оценивается амплитуда и **качество** кадра.
4. **Полосовой фильтр** (Butterworth, полоса в духе 0.8–2.5 Hz, см. `ppg-bandpass.ts` и комментарии в экспорте JSON), затем лёгкое сглаживание MA(3). На детектор пиков идёт сигнал, соответствующий `ppgBandpassed` в снимке.
5. **Пики:** локальные максимумы с порогами по высоте, выступу, отступу от края буфера, **рефрактерному** интервалу (чтобы не считать один пульс дважды).
6. **Параболическая подстройка времени пика:** по трём отсчётам детрендированного сигнала вокруг максимума оценивается смещение вершины в долях шага между кадрами (ограничение ±0.5 отсчёта), перевод в миллисекунды через средний шаг времени между соседними кадрами. Итог записывается как **`timestampMs` принятого удара** (не «целый кадр»).
7. **Слияние с историей:** `mergeBeatTimestampsPhase1` объединяет новые метки с буфером `beatTimestampsMs`, подавляя дубликаты в пределах ~220 ms. История обрезается по `BEAT_HISTORY_WINDOW_MS` (длинная, чтобы «первые 90» валидных ударов не вымывались из merged при длинной сессии).

**RR (между соседними merged-метками):** `RR_i = t[i+1] − t[i]` — используется и для оценки ЧСС, и (через отбор) для HRV.

## 2. Два параллельных мира: «пульс» и «практика HRV»

Это критично для понимания логики:

### 2.1 Окно пульса (ЧСС на экране)

- Скользящее окно по времени (`PULSE_WINDOW_MS`, порядка 10 s) по измерениям RR, отобранным для **пульса** (отдельные правила согласованности ритма, диапазон RR для пульса уже **450–1400 ms** в коде — **уже**, чем HRV 300–2000 ms).
- Отсюда `pulseLockState`: `searching` | `tracking` | `holding` (удержание последней стабильной ЧСС при кратковременном провале).

### 2.2 Практика HRV (RMSSD / стресс)

- После **успешной калибровки пульса** заполняется **отдельный** накопитель **`hrvValidBeatTimestampsMs`**: в него попадают метки ударов, считающихся валидными для HRV (см. ниже).
- Число **`hrvValidBeatCount`** = длина этого накопителя — **не** то же самое, что длина merged-списка или число кадров.
- Именно **`hrvValidBeatCount`** передаётся в `computePracticeHrvMetrics` как `hrvValidBeatCount`, а **`practiceStartMs`** = `hrvAccumulationStartTimestampMs` (момент сброса накопителя при завершении калибровки).

Путаница «почему на экране одно, а в JSON другое» часто из этого: **ЧСС** и **HRV** используют **разные** фильтры RR и **разные** буферы.

## 3. Калибровка пульса (до накопления HRV)

Пока калибровка не завершена:

- Идёт **прогрев** (`WARMING_PHASE_MS`) и окно **стабилизации** (`PULSE_SETTLE_MS`) с проверкой, что в окне достаточно «хороших» кадров с `pulseLockState === "tracking"` и последние RR в допустимых пределах (для проверки используются границы, **согласованные с HRV**: **300–2000 ms** — см. константы из `hrv-practice-constants` / использование `HRV_RR_HARD_*` в анализаторе).
- При провале окно может **перезапускаться** (см. логику `pulseSettleGoodMsAccum` / `validationWindowStartTimestampMs` в коде).

Только после **`pulseCalibrationComplete === true`** начинается накопление **`hrvValidBeatTimestampsMs`** и отсчёт ординалов для метрик практики.

## 4. Два уровня «валидности» удара для HRV

### 4.1 `beatHrvEligible` (по merged-индексу)

Синхронизируется с merged-списком: удар помечается как подходящий для HRV, если в момент фиксации выполнялось **tracking** (не сценарии, где ритм удерживается без полноценного tracking — см. `syncBeatEligibilityFromMerged`). Это отсекает интервалы, полученные в режиме **holding** как ненадёжные для HRV.

### 4.2 Накопитель `hrvValidBeatTimestampsMs`

Пополняется **только после** калибровки. В него добавляются метки ударов из merged, которые одновременно:

- имеют `beatHrvEligible` для этой позиции, и
- попадают под правила добавления в накопитель (см. `appendNewHrvValidBeats` в `finger-analysis.ts` — там зафиксированы условия «новый валидный удар относительно последнего сохранённого»).

**`hrvValidBeatCount`** в формулах = **число элементов** этого накопителя.

### 4.3 Ординалы

`buildEligibleOrdinalSinceStart` присваивает валидным ударам после `practiceStartMs` индексы **0 … N−1**. По ним строятся **префикс** (первые K ударов) и **хвост** (последние K ударов) для сегментов RR.

## 5. Построение ряда RR для метрик практики (`metrics.ts`)

### 5.1 Отбор пар RR

Берутся **соседние** merged-удары, у которых оба индекса **`beatEligible`** (в контексте сбора — это тот же слой eligibility, что связан с HRV; в коде — `beatEligible` в сигнатуре `computePracticeHrvMetrics`).

Для каждой пары дополнительно требуется, чтобы **оба** соседних интервала RR (как пары подряд) удовлетворяли **`rrPairOkForRmssd`**: оба в диапазоне **[300, 2000] ms** (`HRV_RR_HARD_MIN_MS` / `HRV_RR_HARD_MAX_MS`). Это **жёсткий** фильтр до Хампеля.

### 5.2 Префикс и хвост

- **`collectEligibleRrOrdinalPrefix(..., maxOrdinalExclusive)`** — RR между соседними валидными ударами, где **второй** удар имеет ординал **< maxOrdinalExclusive**.  
  Для **«первых 90 ударов»** задаётся `maxOrdinalExclusive = 90` → ординалы второго конца интервалов до 89 включительно → до **89 интервалов RR** (последний заканчивается на 90-м валидном ударе).

- **`collectEligibleRrOrdinalTail(..., tailCount)`** — RR, у которых **оба** конца лежат среди **последних `tailCount`** валидных ударов (формализовано в коде через `minOrd` и верхнюю границу по последнему удару).

### 5.3 Фильтр Хампеля

После фиксации сегмента вызывается **`preparePracticeRr` → `hampelFilterRrIntervals`**:

- окно **13** интервалов;
- порог **3 × MAD × 1.4826**;
- выбросы **заменяются** медианой окна, длина ряда сохраняется.

Далее **все** RMSSD/Баевский по практике считаются **по очищенному** ряду.

## 6. Тиры `computePracticeHrvMetrics` и случай «≥ 90 валидных ударов»

Вход: `hrvValidBeatCount` (= `nBeat`), merged `beatTimestampsMs`, `beatEligible`, `practiceStartMs`.

| Диапазон `nBeat` | Тир | RMSSD / стресс | Начало/конец |
|------------------|-----|----------------|--------------|
| &lt; 30 | `none` | Нет показа | — |
| 30–59 | `beats_30_59` | По всем накопленным RR, **1** блок | Нет |
| 60–89 | `beats_60_89` | **2** блока, приближённые флаги | Нет |
| **90–119** | **`beats_90_119`** | Префикс **ровно первых 90** валидных ударов (сегмент RR до 89 интервалов), **3** блока | Нет |
| 120–179 | `beats_120_179` | Начало: префикс 90; конец: хвост **60**; по **3** и **2** блокам | **Да** |
| 180+ | `beats_180_plus` | Начало: префикс 90; конец: хвост **90**; по **3** и **3** блокам | **Да** |

**Важно для вопроса «90 валидных ударов»:** при **ровно 90–119** валидных ударов используется **только префикс первых 90 ударов** — отдельного «хвостового» сегмента ещё нет; поля `initial*` / `final*` в результате нулевые, `showInitialFinal === false`.

Когда пользователь позже захочет **«один замер за 120 ударов»** (см. `HRV_INTEGRATION_SINGLE_MEASURE_BEATS = 120` в `hrv-practice-constants.ts`), при **120 ≤ nBeat ≤ 179** движок уже переходит в режим **начало/конец** с финалом по **последним 60** ударам — это **не** то же самое, что «ровно 120 ударов в одном сегменте». Для интеграции возможны: (а) читать **`rmssdMs` / сегментный RMSSD из тира `beats_90_119` при достижении 120 без ожидания 180**, или (б) добавить отдельный API, вызывающий ту же математику на явно заданном префиксе длины 120 — это **изменение продукта**, не зашито в текущий тир.

## 7. RMSSD по практике (цепочка RR)

### 7.1 Базовая формула

На ряду RR `RR[0…n−1]` после Хампеля:

\[
\text{RMSSD} = \sqrt{\frac{1}{n-1}\sum_{i=0}^{n-2}(RR_{i+1}-RR_i)^2}
\]

Реализация «классическая»: `computeRmssdStandardFromRrIntervals`.

### 7.2 Усечение (trimmed) для **сегментов** практики

Для **блоков** и запасного варианта по всему сегменту используется **`computeRmssdStandardFromRrIntervalsTrimmed`** (~12% отрезается с каждого конца распределения **квадратов** разностей), чтобы **один-два** артефакта не раздували RMSSD.

### 7.3 Блоки и медиана

Сегмент RR делится на **`B`** равных по длине частей (`splitIntoEqualChunks`). На каждой части считается trimmed RMSSD; итог сегмента — **медиана** по блокам (ненулевые). Если медиана 0 — fallback на trimmed по **всему** сегменту.

### 7.4 Потолок сегмента

`rmssdSegment` ограничивает результат сверху (**`HRV_PRACTICE_RMSSD_ABS_MAX_MS`**, порядка 160 ms), чтобы хвост при ослаблении контакта не давал **200+ ms** при нормальном отображении.

## 8. Индекс Баевского и шкала 0–100

### 8.1 Сырой индекс

На том же сегменте RR (после Хампеля), длина ≥ 3:

1. **Мягкая гистограмма** с шагом 50 ms: каждый RR «делит» вес 1.0 между двумя соседними бакетами (линейно), чтобы значения у границ бакетов не переворачивали моду.
2. Мода: бакет с максимальным суммарным весом → `modeBucketMs`, `modeWeight`.
3. Амплитуда моды **AMo (%)** ≈ `(modeWeight / n) × 100`.
4. Вариационный размах **Δ** = `(max(RR) − min(RR)) / 1000` с, минимум 0.05 с.
5. Сырой индекс: `I_raw = AMo / (2 · T_mode · Δ)` где `T_mode = modeBucketMs / 1000` (`calculateBaevskyStressIndexRaw`).

### 8.2 Проценты

`mapBaevskyStressToPercent`: `min(100, 100·(1 − e^(−I_raw/180)))`.

### 8.3 Блоки

Как и для RMSSD: по блокам считается сырой индекс, затем **медиана** по блокам; fallback на весь сегмент (`stressSegmentRaw`).

## 9. Связка с `finger-analysis.ts`: что видит пользователь

### 9.1 Качество и «надёжный» HRV

- **`qualityOk = signalQuality ≥ HRV_QUALITY_THRESHOLD`** (константа в анализаторе).
- **`hrvUnlocked`** = калибровка завершена ∧ палец обнаружен ∧ уверенность присутствия ≥ порога удержания.
- **`rmssdReady` / `stressReady`** = флаги из `practiceHrv` ∧ **`qualityOk`** (не путать с тем, что математика в `computePracticeHrvMetrics` считается всегда при достаточном `nBeat`).

### 9.2 Обновление «последней хорошей практики»

**`lastPracticeHrvWhileFinger`** обновляется только при **`hrvUnlocked && qualityOk`**, чтобы при ослаблении контакта не сохранялся пересчёт с «испорченным» хвостом RR.

### 9.3 Защёлка начала по **90** валидным ударам

Когда **`hrvValidBeatTimestampsMs.length ≥ HRV_LATCH_INITIAL_AFTER_BEATS`** (90) и латч ещё пуст, однократно сохраняются **`hrvLatchedInitial*`** из текущего `practiceHrv` **при `qualityOk`** — чтобы «начало» не уплывало при росте N.

### 9.4 Отображаемая RMSSD (EMA)

**Не** равна мгновенно «финальному» сегменту: к цели из практики применяется экспоненциальное сглаживание с **постоянной времени** `HRV_RMSSD_DISPLAY_TAU_MS` (~12 s):  
`α = 1 − exp(−Δt/τ)` на кадр. В режиме начало/конец целью для сглаживания берётся **финальный** RMSSD сегмента, если он &gt; 0.

### 9.5 Стресс на экране

- В режиме **начало/конец** в сырой/процентный стресс подставляются **финальные** значения сегмента (без долгого blend к прошлому кадру).
- Иначе — `blendTowards` от предыдущих стабильных значений + удержание при кратковременных провалах (`STRESS_HOLD_MS` и ветка с `pulseLockState`).

### 9.6 Снятие пальца: `hrvSessionEnd*`

При переходе **палец был → палец снят**, при выполнении порогов по числу валидных ударов и наличию **`lastPracticeHrvWhileFinger`**, в поля **`hrvSessionEndInitial*`** / **`hrvSessionEndFinal*`** копируются значения из **последней качественной** практики и латча начала. Сброс при **`resetMetricWindows`**.

**Замечание для проверки:** «живые» поля **`hrvFinalRmssdMs` / `stressIndex`** в снимке на каждом кадре отражают **текущий** `practiceHrv`; при плохом качестве отображение может **не** догнать цель (EMA/ветки удержания), тогда как «сырой» финал в объекте практики мог обновиться — это **не баг математики**, а разделение слоёв.

## 10. Известные ограничения и места будущих ошибок

1. **Два расчётных пути:** `computePracticeHrvMetrics` (фактически используется) vs `updateHrvMetrics` (не подключён) — при рефакторинге не перепутать.
2. **Смена тира** (например 119→120) меняет определение сегментов → возможны **скачки** «финала».
3. **Баевский на PPG** завышает проценты при узком RR и малом Δ — субъективно «высокий стресс в покое»; это свойство формулы на коротком окне, не только баг.
4. **Интеграция на 120 ударов:** используйте константу **`HRV_INTEGRATION_SINGLE_MEASURE_BEATS`** и явно решите, нужен ли отдельный вызов метрик по фиксированному префиксу 120, или достаточно тира **`beats_120_179`** / ожидания **180+**.
