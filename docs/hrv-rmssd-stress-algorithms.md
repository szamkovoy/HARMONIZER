# HRV по пальцевому PPG: полный алгоритм RMSSD и индекса стресса (Баевский)

Документ описывает **фактическую** реализацию в репозитории на момент последней правки. Зафиксированы: **два буфера ударов**, **тиры по числу валидных ударов** в `hrvValidBeatTimestampsMs`, **различие Хампеля для RMSSD и для Баевского**, и **как считается то, что видит пользователь** (EMA, удержание).

**Код:** `modules/biofeedback/core/finger-analysis.ts`, `modules/biofeedback/core/metrics.ts`, `modules/biofeedback/core/hrv-practice-constants.ts`, `modules/biofeedback/core/ppg-bandpass.ts`, пробы: `modules/biofeedback/ui/BiofeedbackProbeScreen.tsx`. Интеграция с мандалой: `modules/biofeedback/core/mandala-adapter.ts`.

## 1. Вход: от кадра камеры до точки анализа

1. Оптический индекс: `redMean − 0.35·greenMean − 0.15·blueMean`.
2. Детренд, качество, полосовой фильтр (см. `ppg-bandpass.ts`), MA(3) на детектор пиков.
3. Пики, параболическая подстройка времени, merge с историей (`mergeBeatTimestampsPhase1`), подавление дубликатов ~220 ms. История merged обрезается по `BEAT_HISTORY_WINDOW_MS` (длинная, для стабильности детектора).

**Важно:** merged-список нужен **пульсу** и eligibility; **метрики практики HRV** считаются по **отдельному накопителю** после калибровки (§2.2).

---

## 2. Два параллельных мира: «пульс» и «практика HRV»

### 2.1 Окно пульса (ЧСС)

Скользящее окно по RR для пульса (границы **450–1400 ms** и др.). `pulseLockState`: `searching` | `tracking` | `holding`.

### 2.2 Практика HRV (RMSSD / стресс)

После калибровки заполняется `**hrvValidBeatTimestampsMs`**: дедуплицированные метки валидных для HRV ударов (`appendNewHrvValidBeats`). Число элементов = `**hrvValidBeatCount**` — это **единственный** источник длины ряда для `computePracticeHrvMetrics` и для диагностического `computePracticeRmssdHampelDiagnostics`.

`practiceStartMs` / `hrvAccumulationStartTimestampMs` задают момент сброса накопителя при завершении калибровки; в накопитель попадают только удары после этого (см. код анализатора).

---

## 3. Калибровка пульса

Прогрев `WARMING_PHASE_MS`, окно стабилизации `PULSE_SETTLE_MS`, проверка tracking и RR в диапазоне **300–2000 ms** для успеха калибровки. До `pulseCalibrationComplete` накопитель HRV не растёт.

---

## 4. Валидность удара для HRV

- `**beatHrvEligible`** на merged-индексе: синхронизация с tracking (`syncBeatEligibilityFromMerged`).
- `**hrvValidBeatTimestampsMs**`: подмножество с особыми правилами «новый удар относительно последнего сохранённого» (см. `appendNewHrvValidBeats`).

---

## 5. Построение ряда RR для метрик практики (`metrics.ts`)

### 5.1 Источник

Берётся **строго упорядоченный** массив `hrvValidBeatTimestampsMs` (монотонные времена ударов). Для сегмента «первые K ударов» или «последние K ударов» вырезается окно по **индексам ударов**, затем для каждой **соседней пары** в этом окне считается `RR = t[i+1] − t[i]`.

### 5.2 Жёсткий фильтр

В ряд попадают только интервалы с **300–2000 ms** (`HRV_RR_HARD_*`). Пары, не прошедшие диапазон, **пропускаются** (в массив RR не попадают).

Реализация: `collectRrFromBeatWindow` / `collectRrFromBeatPrefixBeats` / `collectRrFromBeatTailBeats` (внутренние хелперы в `metrics.ts`).

### 5.3 Хампель: два режима

- **Баевский и любые расчёты по «полному ряду фиксированной длины»:** `preparePracticeRr` → `hampelFilterRrIntervals`: выбросы **заменяются** медианой окна (13 интервалов, `HRV_HAMPEL_NSIGMA` = 3). Длина ряда сохраняется.
- **RMSSD в продакшене:** `hampelOutlierFlags` на **сыром** жёстко отфильтрованном ряду; интервалы, помеченные как выбросы, **не подставляются** медианой в RMSSD — из суммы квадратов **соседних разностей** исключаются пары, где **хотя бы один** из двух соседних RR помечен выбросом. Далее — trimmed RMSSD, медиана по блокам, потолок `HRV_PRACTICE_RMSSD_ABS_MAX_MS` (~160 ms).

Таким образом, экспертная рекомендация «для RMSSD не залепливать дыры медианой» учтена; Баевский остаётся на сглаженном ряду.

## 6. Тиры `computePracticeHrvMetrics`

Вход: длина `**hrvValidBeatTimestampsMs`** = `nBeat`.


| Диапазон `nBeat` | Тир              | RMSSD / стресс                                | Начало/конец |
| ---------------- | ---------------- | --------------------------------------------- | ------------ |
| < 30             | `none`           | Нет показа                                    | —            |
| 30–59            | `beats_30_59`    | Префикс всех накопленных ударов, 1 блок       | Нет          |
| 60–89            | `beats_60_89`    | 2 блока, приближённые флаги                   | Нет          |
| 90–119           | `beats_90_119`   | Префикс **первых 90** ударов, 3 блока         | Нет          |
| 120–179          | `beats_120_179`  | Начало: первые 90; конец: хвост **60** ударов | Да           |
| 180+             | `beats_180_plus` | Начало: первые 90; конец: хвост **90**        | Да           |


Поле `**rmssdMs`** в результате соответствует **финальному** сегменту (хвост), когда включён режим начало/конец.

---

## 7. RMSSD по практике

Формула классического RMSSD по ряду RR: корень из среднего квадратов **последовательных** разностей соседних интервалов.

На сегменте: после исключения выбросов для пар — **trimmed** по квадратам разностей (~12%), при нескольких блоках — **медиана** trimmed RMSSD по блокам, fallback, затем потолок.

---

## 8. Индекс Баевского и шкала 0–100

На сегменте после **импутации** Хампелем (как в §5.3): мягкая гистограмма с шагом 50 ms, мода, AMo, Δ, сырой `I_raw` (`calculateBaevskyStressIndexRaw`), блоки — медиана по блокам.

Нормировка в 0–100: `mapBaevskyStressToPercent` — `100·(1 − e^(−I_raw / D))`, с `**D = BAEVSKY_STRESS_PERCENT_DIVISOR`** (в коде **220**, мягче, чем старые 180, чтобы реже «прилипать» к верху на PPG).

## 9. Связка с `finger-analysis.ts`: что видит пользователь

### 9.1 Качество

`qualityOk = signalQuality ≥ HRV_QUALITY_THRESHOLD`. `rmssdReady` / `stressReady` = флаги из `practiceHrv` ∧ `qualityOk`.

### 9.2 Отображаемая RMSSD (EMA)

К цели из практики применяется EMA с τ = `HRV_RMSSD_DISPLAY_TAU_MS` (~12 s). В режиме начало/конец цель — **финальный** RMSSD сегмента (если > 0).

### 9.3 Отображаемый стресс

- **Без** начало/конец: `blendTowards` к `practiceHrv.stressPercent` с коэффициентом **0.16** после первого кадра (мягкое следование).
- **С** начало/концом: цель — **финальный** процент Баевского по хвосту; кадр к кадру применяется **EMA** с τ = `HRV_STRESS_DISPLAY_TAU_MS` (~12 s) к той же цели (как у RMSSD), а не мгновенная подстановка. Это снимает резкий скачок при смене тира и делает **микродвижения** при медленном сдвиге хвоста заметнее.

**Плато одного и того же числа (например 40.1 на экране)** на длинном участке при **стабильном** целевом `finalStressPercent` и отсутствии новых валидных ударов (хвост из 90 ударов в миллисекундах не меняется) — **ожидаемо**: метрика обновляется при **новом** ударе в накопителе. Пока длина/состав хвоста в терминах времён не меняются, сырой расчёт может быть **бит-в-бит** неизменным десятки секунд.

Удержание при провале качества: `STRESS_HOLD_MS` (~12 s) и ветка с `pulseLockState`.

### 9.4 Снятие пальца: `hrvSessionEnd*`

При переходе палец был → палец снят копируются последние качественные значения и латч начала.

---

## 10. Известные ограничения

1. Смена тира (например 119→120) меняет определение сегментов → возможен скачок цели; EMA смягчает отображение.
2. Баевский на PPG может субъективно завышать «стресс» при узком RR — свойство формулы на коротком окне.
3. `updateHrvMetrics` не подключён к камере — не путать с `computePracticeHrvMetrics`.

## 11. Диагностика RMSSD (`computePracticeRmssdHampelDiagnostics`)

- Вход: `**hrvValidBeatTimestampsMs`** (как основные метрики).
- `**schemaVersion`: 2** в экспортируемом JSON.
- На том же сегменте, что и `rmssdMs`: `**rmssdClassicNoHampelMs`** (без Хампеля), `**rmssdPipelineMs**` (как в продакшене — исключение выбросов для RMSSD + trimmed + блоки + потолок).
- В сегменте: `**rrMsAfterHampel**` — ряд с **импутацией** (для согласованности с Баевским и наглядности); `**hampelOutlierMask`** — какие позиции считались выбросами.
- Кнопка в Biofeedback Probe: «Экспорт JSON: RMSSD классика vs пайплайн». На **iOS** для `Share` используется URL вида `file://…` для файла в кэше.

Критерий «разница > ~30% между классикой и пайплайном» — эвристика; на очень коротких рядах RR после жёсткого фильтра сравнение шумное.
