# Стандарт документации модулей Harmonizer

Все функциональные модули системы должны иметь файл `readme.md` в корне модуля (`modules/<имя>/readme.md`), а при необходимости — краткий указатель или дорожную карту в `/docs/modules/`.

Общая архитектура описана в `[system_structure.md](system_structure.md)`: UI на **React Native**, вызовы к внешним API через `**services/`**, серверные ключи — только на **Vercel** (`_legacy_web` или аналог).

## Шаблон описания модуля:

1. **Назначение**: Краткое описание цели модуля (какую задачу решает).
2. **Интерфейсы (Inputs/Outputs)**:
  - Какие параметры принимает (Props, аргументы функций).
  - Что возвращает (события, данные).
3. **Внешние зависимости**: HTTP API (Gemini через прокси), Hume, Supabase; нативные модули Expo при необходимости (`expo-av` и т.д.).
4. **Логика работы**: Ключевые алгоритмы и связь с `services/`.
5. **План развития**: Улучшения (Live API, биометрия, офлайн и т.д. — см. `[roadmap.md](roadmap.md)`).

*Документацию модуля следует обновлять при изменении публичных интерфейсов или контрактов с API.*
